1. Edit-before-confirm AI log loop · effort M

YELLOW

What: Build the correction-loop UX for any AI-suggested log entry (photo, voice, or text). When the user submits a meal photo or Hebrew voice description, the agent returns a card (not a committed entry) showing each detected item as an editable chip: ingredient name (Hebrew), portion (g), confidence %. User can tap to swap from a 3-item suggestion list, drag a slider for portion ±50%, long-press to delete, or tap + הוסף פריט to add a missed item. Confirm button is greyed until the user has looked at the card (3-second dwell required). Every correction writes a (predicted, corrected) pair to a per-user disambiguation log that biases the next prediction toward the user's pantry. Hebrew label: בדוק לפני שאישרת. AI proposes, user disposes.

Inspired by
Welling.ai 2026 correction-loop commentary; Lifesum cashew-as-shrimp accuracy gap; Lose It "Snap It" without edit affordance; Tubik 2026 "AI as copilot, not autopilot."
Why it matters
This is the single feature the entire category has shipped badly. Owning the correction loop in Hebrew — where Israeli ingredients (לאבן, טחינה גולמית, חציל בלאדי) are guaranteed to confuse a US-trained vision model — turns our biggest AI vulnerability into a wedge. Also gives Agent 4 (Inventory) a continuously improving signal it doesn't have today. For Neo's unit-economics frame: correction time per meal is the retention KPI; publish it once we beat 8s.
User segment
All AI-logging users (~40% of active users within 60 days of any AI-log launch); strongest impact on home cooks logging composite Israeli dishes.
Tier reason
New screen + new agent behavior + small per-user log table. Vision API likely already in scope from prior AI work. Feature-flagged. No PII beyond what we already store.
